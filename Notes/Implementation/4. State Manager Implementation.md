### Step 1: Start PostgreSQL locally via Docker 

```
  docker run -d \
    --name migration-postgres \
    -e POSTGRES_USER=admin \
    -e POSTGRES_PASSWORD=changeme \
    -e POSTGRES_DB=migration_orchestrator \

Verify it is running:
```
  docker ps | grep migration-postgres

### Step 4: Set up states.py

- States.py is a python file defined as an enum class that contains all states that a server can take 
- This is defined as an Enum class so that the state can be directly referenced as a constant rather than a string. 
```
  PENDING = "PENDING"
  STATES.PENDING can be referenced and the string "PENDING" can be used to append to the database rather than having a python object.
```
- The states are mapped into four categories which are defined as frozensets. These frozen sets are configured to handle decisions (such as automating jobs/rollback) without having to write long if conditionals. 
  STATES:
  - ACTIVE
  - IN PROGRESS
  - COMPLETED
  - ERROR

### Step 5: Set up transitions.py

- This python module helps define set of all valid transactions that a server can make. 
- In case any invalid transactions are made on accident, an exception is thrown to ensure database consistency with the state flow. 
- This also make sure that completed or error states. Failed/frozen jobs will not proceed without human intervention, to prevent the automated system from blindly retrying the job as it could fail again. 
- Here is the full flow of the python module:
```
  1. Is from_state a word we recognise? No → reject.
  2. Is to_state a word we recognise? No → reject.
  3. Is the server already finished (completed or error)? Yes → reject.
  4. Is this exact from → to pair on the approved list? No → reject.
  5. All checks passed → return silently, allow the database write to proceed.
```

### Step 8: Implement the state manager 
- The state manager class is the main and only module that directly writes updates to the database. It imports the transition validator to check validity before writing to the database. 
- Row locking is implementing using the 'SELECT FOR UPDATE' query.
- create is defined as a class method. This is because init cannot be initialised with await. Since creation of servers need to happen simultaneously, this is implemented as a class method. 
- self.pool holds the asyncpg connection pool - multiple connections. This allows multiple open tcp connections to postgres. 
-   Opening a TCP connection to PostgreSQL every time a job runs is expensive — TLS handshake, authentication, setup — this can take tens of milliseconds. If you have 50 jobs running per second, that overhead adds up fast.

	  The pool solves this by keeping connections open and reusing them:
	  Job arrives → pool.acquire() → borrow an existing open connection
	  Job finishes → pool releases → connection returned, stays open for next job
