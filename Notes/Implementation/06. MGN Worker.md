#### MGN worker is the component that directly talks to AWS. It is the only component that handles all the AWS calls -> from adding a source server for migration to the final cutover. It consumes the mgn_jobs queue held by the RabbitMQ statefulset. 

## Flow:
```
  RabbitMQ delivers message
          ↓
  Parse: job_type, server_id, aws_source_server_id, payload
          ↓
  Check StateManager: is the server still in the expected state?
  (if not, message is stale — ack and skip)
          ↓
  Call the appropriate handler
          ↓
  Handler makes the real AWS API call (MGN or EC2)
          ↓
    ┌─── Success ──────────────────────────────────────────────────────┐
    │  advance server state in PostgreSQL                              │
    │  dispatch follow-up job if needed (e.g. POLL_REPLICATION_STATUS) │
    │  ack the message                                                 │
    └──────────────────────────────────────────────────────────────────┘
    ┌─── Failed ───────────────────────────────────────────────────────┐
    │  nack(requeue=False) → message goes to DLX → Rollback Worker    │
    └──────────────────────────────────────────────────────────────────┘
```


## Components:
### mgn_result.py:
- This works in the same way the poller_result.py worked. 
- Every handler deals with one of two things:
```
    MgnResult.success(next_state="AGENT_INSTALLED", metadata={"aws_source_server_id": "s-0abc"})
  MgnResult.failed(error="MGN API timeout after 30s")
```

- This reads the result and decides what to do next with respect to the rabbitmq queue, whether that is to dispatch a job/advance state and so on. The handlers only deal with the AWS API. 

### aws_result.py:
- This is used to create a boto3 client, which is an aws python sdk -> basically how python communicates with aws api 
- This is set up as a separate file to initialise the boto3 client only once for all the handlers to use rather than re-initialising it over and over again. 
- Two clients exist:
```
  - MGN client — talks to mgn.{region}.amazonaws.com. Knows about source servers, replication configurations, launch templates, test launches, cutover. This is the Migration Service.

  - EC2 client — talks to ec2.{region}.amazonaws.com. Knows about instances, their states, terminating them. After MGN launches a test or cutover instance, that instance is just a regular EC2 instance. (now omitted)
```

### mgn_worker.py:
 - This is the main worker class, identical to poller_worker.py
 - This implements all the AWS API calls, is used for dispatching the next job (whether that's poll/rollback/approval).
 - This uses the mgn_client defined in aws_result to call AWS APIs in a concurrent manner (using async-await)
 - #connection variable is used to connect to rabbitmq via the amqp url (whose port had already been defined and opened up when setting up rabbitmq. This is how dispatchers and workers connect to the queue to consume/dispatch jobs)
 - #channel variable is used to declare a certain queue. This indicates "this is the queue/channel we will be consuming messages from"
```
   queue = await self._channel.declare_queue(POLL_QUEUE, passive=True)
```

### run_mgn.py:
- Entry-point to the mgn worker. This creates the StateManagerClient, mgn worker, connects it to rabbitmq

#### In addition to the mgn_worker.py implementation, 5 poll handlers were also additionally added to poller_worker.py as the mgn client has been initialized.
