#### This is the only external interface to the entire system. It allows for registering servers and kicking off the pipeline, takes care of the actions succeeding human approval gates, server and state transition history tables for visibility

### Dispatches it takes care of:
- Next steps of the pipeline are triggered by some mgn and poll jobs, but not all. Suppose if there is human approval required for a state, the pipeline gets stalled. The API is what manages these steps to make sure the pipeline doesn't get stalled.
![](../Images/Pasted%20image%2020260313122433.png)


### Approval flow:
  1. Read server from DB, validate it is in an AWAITING_* state
  2. Determine the approved next state from the transitions table
  3. Call state_manager.advance_state() with triggered_by=engineer_id (not "system" — human action gets the engineer's ID in the audit trail)
  4. Look up what job to dispatch for the new state
  5. Dispatch via JobDispatcher
  6. Return new state

### Reject flow:
  1. Validate server is in an AWAITING_* state
  2. Advance state directly to FAILED with rejection reason in metadata
  3. No automatic AWS cleanup is done.

### Key features:
  **Startup** **(lifespan)**: Creates StateManager (asyncpg pool) and JobDispatcher (RabbitMQ connection). Both are stored on app.state and injected into routes via FastAPI dependency injection. Both are cleanly closed on shutdown.

  **Pipeline** **enforcement** **at** **the** **API** **layer**: Every action endpoint reads the server's current_state from the DB before doing anything. If the server isn't in the expected state, it returns 409 Conflict with a message telling the engineer what state the server is currently in and what's required. This is immediate feedback — the engineer doesn't have to wait for a worker to fail and roll back. The state machine then enforces the same rules again at the DB level (defence in depth).

  **Action** **endpoints** **(25** total, excluding batch endpoints):**

  - POST /servers — registers a new server at PENDING. Body includes aws_account_id and aws_region (stored on the server record — credentials themselves stay in the k8s secret).

  - POST /servers/{id}/start — dispatches ADD_SERVER. Requires aws_source_server_id (the MGN-assigned ID). Server must be in PENDING.

  - POST /servers/{id}/configure-replication — dispatches CONFIGURE_REPLICATION_SETTINGS. Server must be in AGENT_INSTALLED.

  - POST /servers/{id}/configure-test-launch — dispatches CONFIGURE_TEST_LAUNCH_TEMPLATE. Server must be in READY_FOR_TESTING.

  - POST /servers/{id}/configure-cutover-launch — dispatches CONFIGURE_CUTOVER_LAUNCH_TEMPLATE. Server must be in CUTOVER_LAUNCH_TEMPLATE_CONFIGURED.

  - **7** **gate-specific** **approve** **endpoints** (approve-replication, approve-test-launch, approve-test-validation, approve-cutover-launch, approve-cutover-validation, approve-archive, approve-cleanup) — each validates the server is in the specific AWAITING_* state, calls advance_state() with triggered_by=engineer_id, and dispatches the appropriate next job.

  - **7** **gate-specific** **reject** **endpoints** — each validates the server is in the specific AWAITING_* state, calls advance_state() to FAILED with triggered_by=engineer_id.

  - POST /servers/{id}/reset — calls StateManager.reset_server() (bypasses TransitionValidator). Only valid for FAILED state.



  **Query** **endpoints**: GET /servers (list, optional ?state= filter), GET /servers/{id}, GET /servers/{id}/history.


  **Why** **gate-specific** **approve/reject** **instead** **of** **a** **generic** **/approve**: Each gate is a semantically different human decision — approving replication security group config is nothing like approving a live production cutover. Gate-specific endpoints give the engineer an unambiguous action, return 409 if called at the wrong point in the pipeline, and produce a clear audit trail in state_transition_history

## Additional Changes — Batch Support

### What is a Batch?

A batch (also called a "wave") is a named group of servers that move through the migration pipeline together. It allows an engineer to trigger the same pipeline action — configure replication, approve test launch, approve cutover — across all servers in the batch with a single API call, instead of calling each server endpoint individually.

A server belongs to at most one batch at a time. The relationship is tracked via a nullable `batch_id` foreign key on the `servers` table — `NULL` means the server is being migrated individually with no batch context. One server, one batch; no many-to-many join table. Deleting a batch does not delete its servers — it clears their `batch_id` via `ON DELETE SET NULL` so they can be reassigned or migrated individually.

*`batches` table columns:*
| Column | Type | Notes |
|--------|------|-------|
| `batch_id` | VARCHAR (PK) | Engineer-supplied, human-readable identifier |
| `name` | VARCHAR NOT NULL | Display name for the wave |
| `description` | VARCHAR (nullable) | Optional longer description |

*Changes to `servers` table:*
- New nullable `batch_id VARCHAR` column with FK → `batches.batch_id` (`ON DELETE SET NULL`)
- New index `ix_servers_batch_id` on `servers.batch_id` (dominant query pattern is "all servers in batch X")

The three JSONB config columns on `batches` serve as an audit record: they capture exactly what configuration the entire wave used at each configure step, providing a reference for any future investigation or catch-up work.

### New StateManager Methods

The following methods were added to `StateManager` to support batch operations:

**Updated existing methods:**
- `register_server()` — new optional `batch_id` param; includes the value in the INSERT so a server can be assigned to a batch at registration time.
- `list_servers()` — replaced single `state_filter` arg with four composable optional filters: `state_filter`, `batch_id`, `assigned_engineer`, `hostname`. Uses a dynamic parameterised WHERE clause. `batch_id='unassigned'` is a sentinel that generates `WHERE batch_id IS NULL`.

**New batch management methods:**
- `create_batch(batch_id, name, created_by, description)` — inserts a new batch record. Raises `asyncpg.UniqueViolationError` on duplicate `batch_id`.
- `get_batch(batch_id)` — returns the batch record with an additional `server_count` field computed via a `LEFT JOIN COUNT` on `servers`. Raises `ValueError` if not found.
- `list_batches()` — returns all batches with server counts, newest first.
- `add_servers_to_batch(batch_id, server_ids)` — bulk-updates `servers.batch_id`. Returns three lists: `assigned` (newly moved), `already_assigned` (already in this batch), `not_found` (server_id not in the DB). Works for both NULL → batch and other_batch → this batch.
- `get_batch_servers(batch_id)` — returns all server rows with `batch_id = $1`.
- `get_batch_history(batch_id)` — returns all transitions for every server in the batch via a JOIN on `servers`. Raises `ValueError` if batch not found.
- `update_batch_config(batch_id, config_type, config)` — stores a JSONB config snapshot on the batch record. `config_type` must be one of `replication_config`, `test_launch_config`, `cutover_launch_config`.
- `list_transitions(...)` — global history query with eight optional filters: `server_id`, `batch_id`, `from_state`, `to_state`, `triggered_by`, `job_type`, `since`, `until`. When `batch_id` is provided, the query joins to `servers` to filter by wave. Returns newest-first.

### New and Updated API Routes

**`GET /servers`** — updated to accept four optional query params: `state`, `batch_id`, `assigned_engineer`, `hostname`. All are combinable. `batch_id=unassigned` queries servers not in any batch.

**`POST /servers`** — updated to accept an optional `batch_id` field. If provided, the server is assigned to that batch at registration time.

**`POST /servers/bulk`** (HTTP 207) — bulk register multiple servers in one request. Each item in the `servers` list is a full registration payload. Returns per-server `succeeded`/`failed` results. A single failure does not abort the others.

**`POST /servers/bulk-start`** (HTTP 207) — bulk dispatch `ADD_SERVER` for multiple servers. Each item needs `server_id` and `aws_source_server_id`. Each server must be in `PENDING`. Returns per-server results.

**`GET /history`** — global transition history with eight composable query filters (server_id, batch_id, from_state, to_state, triggered_by, job_type, since, until). Returns newest-first.

**`POST /batches`** — create a new batch. Returns 409 on duplicate `batch_id`.

**`GET /batches`** — list all batches with server counts.

**`GET /batches/{batch_id}`** — get single batch including config snapshots and server count.

**`POST /batches/{batch_id}/servers`** — assign servers to a batch. Returns `assigned`, `already_assigned`, `not_found` lists.

**`GET /batches/{batch_id}/servers`** — list all servers in the batch.

**`GET /batches/{batch_id}/history`** — full audit trail for all servers in the batch.

**Batch pipeline action endpoints (17 total)** — one endpoint per pipeline action across the five stages. Each iterates all servers in the batch, skips any not in the required state, and applies the action to the rest:
- Stage 2: `configure-replication`, `approve-replication`, `reject-replication`
- Stage 3: `configure-test-launch`, `approve-test-launch`, `reject-test-launch`, `approve-test-validation`, `reject-test-validation`
- Stage 4: `configure-cutover-launch`, `approve-cutover-launch`, `reject-cutover-launch`, `approve-cutover-validation`, `reject-cutover-validation`
- Stage 5: `approve-archive`, `reject-archive`, `approve-cleanup`, `reject-cleanup`

All batch pipeline action responses contain three lists: `succeeded` (state advanced and job dispatched), `skipped` (server not in required state — not an error), `failed` (exception during advance or dispatch).

The three configure endpoints (`configure-replication`, `configure-test-launch`, `configure-cutover-launch`) additionally call `update_batch_config()` to store the config snapshot on the batch record before dispatching jobs to individual servers.
