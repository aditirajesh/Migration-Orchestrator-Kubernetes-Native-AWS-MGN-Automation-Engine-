#### A job dispatcher is required to actually ship the messages to the queues for the workers to consume and execute them. This needs to properly communicate to the RabbitMQ statefulset via AMQP

- All the different stages and jobs defined within will be tagged to a certain queue for processing. 
- Make sure that the job types are as loosely coupled as possible. This will prevent rollbacks from undoing un-erroneous steps

### 2. Create the Job Dispatcher
- This job dispatcher will be used to dispatch the jobs into the queue
- #connect_robust is used to automatically reconnect to the broker in case the connection is lost (a regular connect will not restore a broken connection)
- #publisher_confirms is used to communicate back to publisher that the message has been obtained by the broker. This is set up as an async await function to make sure the transaction is complete only when an ack is obtained from the broker 
- In this case, the default/nameless exchange is used where the exchange routes the message when the routing key = queue name. 
- #DeliveryModePersistent is set to ensure that the message is written to the disk, not just memory. Both a durable queue and a persistent delivery mode is required to ensure nothing is lost on restart. 
- The dispatch connects via the AMQP port (5672) to connect with rabbitmq. Then, via this port, the dispatcher will start sending jobs to the queue. 

### 3. Test the dispatch 
- test_dispatch.py is used to check if all the basic plumbing works - if the messages are being properly sent to the correct queues, if it appears in the management UI, and so on. This is a safety measure before directly testing out the dispatch in the kubernetes env with fastapi
- Port-forward the 15627 port to access the management UI to see if the messages are showing up AND also run the AMQP port as this is the port the dispatcher connects to
```
# Terminal 1 — AMQP port (what the dispatcher connects to)
  kubectl port-forward svc/rabbitmq 5672:5672 -n [namespace]

  # Terminal 2 — Management UI (to watch messages arrive)
  kubectl port-forward svc/rabbitmq 15672:15672 -n [namespace]

  # Terminal 3 — run the test
  cd [project-path]
  pip install -r requirements.txt
  python3 test_dispatch.py
```


### 4. Tackling duplicate messages issue:
- RabbitMQ guarantees that messages will be sent 'atleast once' not 'exactly once'. That means that there is a possibility of duplicate messages to be a part of the queue. The effects can range from harmless to catastrophic depending on the type of job. 
- ![[img6.png]]
- To mitigate this: every message gets a unique job_id, and the workers would have to check the job_id against de-duplication before beginning the execution of a task. 
- However, the uuid approach will only resolve the RabbitMQ redelivery issues, which might not be the reason for all duplicated messages. 
- This is why the worker will have to be configured in such a way that it checks with the state manager before the execution of jobs to make sure they are in line. 
- The worker handles de-duplication checks in place of the queue, since any additional responsibility/misconfiguration of the queue could lead to messages being silently dropped.
