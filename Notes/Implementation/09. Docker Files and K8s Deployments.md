## Why Dockerfiles Are Needed

Each component of the system runs as a separate process. Docker packages each process with its Python dependencies and source code into a self-contained image. Kubernetes then runs those images as pods inside the kind cluster. Without images, Kubernetes has nothing to schedule.

---

## Two Dockerfiles

### `docker/Dockerfile.workers`

Shared by all three workers (mgn, poller, rollback) and the one-time Alembic migration job. One image, different run commands per deployment.

```dockerfile
FROM python:3.12-slim

WORKDIR /app/src

COPY src/requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY src/ ./

CMD ["python", "-m", "workers.run_mgn"]
```

**Why `WORKDIR /app/src`:**
Every run file (`run_mgn.py`, `run_poller.py`, `run_rollback.py`) starts with `sys.path.insert(0, ".")`. This adds the current working directory to Python's module search path, which is how imports like `from workers.mgn_worker import MgnWorker` and `from state_manager.state_manager import StateManager` resolve. Setting WORKDIR to `/app/src` means `.` is the `src/` directory at runtime, so all packages are visible.

**Why requirements.txt is copied first:**
Docker builds in layers. Copying and installing requirements before copying source code means the dependency layer is cached. Rebuilds caused by source code changes (the common case) skip the slow `pip install` step entirely.

**Why one shared image:**
All three workers are written in the same codebase, share the same dependencies, and run from the same `src/` directory. A single image keeps build time and storage minimal. Each k8s deployment simply overrides the `command` field:
- mgn-worker: `python -m workers.run_mgn`
- poller-worker: `python -m workers.run_poller`
- rollback-worker: `python -m workers.run_rollback`
- db-migrate job: `alembic upgrade head`

---

### `docker/Dockerfile.api`

Separate image for the FastAPI service. Identical structure to the workers image but exposes port 8000 and defaults to the API entry point.

```dockerfile
FROM python:3.12-slim

WORKDIR /app/src

COPY src/requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY src/ ./

EXPOSE 8000

CMD ["python", "-m", "api.run_api"]
```

The API is kept as a separate image so workers and API can be built and versioned independently. A change to the API does not force a rebuild of the workers image.

## K8s Manifests

```
k8s/
  kind-cluster.yaml                    — Cluster definition: 1 control-plane + 2 workers.
                                         extraPortMappings maps NodePort 30800 → localhost:8000.

  namespace.yaml                       — migration-orchestrator namespace

  aws-credentials/
    secret.yaml                        — AWS STS credentials (REPLACE_ME placeholders).
                                         Populated by `kubectl create secret` before demo run.
                                         Read by all three worker deployments.

  iam/
    mgn-worker-policy.json             — IAM policy document (reference only, not applied by kubectl).
                                         Defines the minimum MGN + EC2 permissions required.

  postgres/
    secret.yaml                        — username / password / database
    statefulset.yaml                   — Single PostgreSQL pod with PersistentVolumeClaim
    service.yaml                       — ClusterIP: postgres.migration-orchestrator.svc.cluster.local:5432
    headless-service.yaml              — Required by StatefulSet for stable DNS per pod

  rabbitmq/
    secret.yaml                        — username / password / erlang-cookie
    configmap.yaml                     — rabbitmq.conf + definitions.json (queues, DLX, bindings)
    statefulset.yaml                   — Single RabbitMQ pod with PersistentVolumeClaim
    service.yaml                       — ClusterIP: rabbitmq.migration-orchestrator.svc.cluster.local:5672
    headless-service.yaml              — Required by StatefulSet

  db-migrate/
    job.yaml                           — One-time Kubernetes Job. Runs `alembic upgrade head`.
                                         Uses workers image. DATABASE_URL uses postgresql://
                                         (synchronous psycopg2 — what Alembic's env.py expects).
                                         backoffLimit: 3, restartPolicy: Never.

  workers/
    mgn-worker/deployment.yaml         — 2 replicas. Calls AWS MGN + EC2 APIs.
                                         Injects AWS credentials from aws-credentials secret.
    poller-worker/deployment.yaml      — 2 replicas. Polls MGN jobs and EC2 reachability.
                                         Injects AWS credentials (EC2 calls added for 2/2 checks).
    rollback-worker/deployment.yaml    — 1 replica (serialised rollback). Executes undo actions.
                                         Injects AWS credentials.

  api/
    deployment.yaml                    — 1 replica. FastAPI + uvicorn on port 8000.
                                         No AWS credentials — only talks to RabbitMQ + PostgreSQL.
    service.yaml                       — NodePort 30800. Exposes API on the cluster.
```

## Credential Flow

All credentials stay in Kubernetes Secrets and are never baked into images.

| Secret | Keys | Consumed by |
|--------|------|-------------|
| `postgres-secret` | username, password, database | db-migrate job, all workers, API |
| `rabbitmq-secret` | username, password, erlang-cookie | all workers, API |
| `aws-credentials` | AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN, AWS_DEFAULT_REGION | mgn-worker, poller-worker, rollback-worker |

Each deployment pulls the individual keys from the relevant secret into named env vars, then composes the full connection URLs using Kubernetes env var substitution (`$(VAR_NAME)`):

```yaml
- name: DATABASE_URL
  value: "postgresql+asyncpg://$(PG_USER):$(PG_PASSWORD)@postgres...5432/$(PG_DATABASE)"

- name: AMQP_URL
  value: "amqp://$(RABBIT_USER):$(RABBIT_PASSWORD)@rabbitmq...5672/"
```

The Alembic migration job uses `postgresql://` (synchronous psycopg2) instead of `postgresql+asyncpg://` because Alembic runs synchronous migrations.

## Build and Deploy Sequence

```bash
# 1. Build images (run from project root)
docker build -f docker/Dockerfile.workers -t migration-orchestrator/workers:latest .
docker build -f docker/Dockerfile.api     -t migration-orchestrator/api:latest .

# 2. Load images into the kind cluster (no registry needed)
kind load docker-image migration-orchestrator/workers:latest --name migration-orchestrator
kind load docker-image migration-orchestrator/api:latest     --name migration-orchestrator

# 3. Apply namespace + infrastructure
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/postgres/
kubectl apply -f k8s/rabbitmq/

# 4. Inject AWS credentials
kubectl apply -f k8s/aws-credentials/secret.yaml   # after replacing REPLACE_ME values

# 5. Run migrations (wait for job to complete before starting workers)
kubectl apply -f k8s/db-migrate/job.yaml
kubectl wait --for=condition=complete job/db-migrate -n migration-orchestrator --timeout=60s

# 6. Deploy workers and API
kubectl apply -f k8s/workers/mgn-worker/
kubectl apply -f k8s/workers/poller-worker/
kubectl apply -f k8s/workers/rollback-worker/
kubectl apply -f k8s/api/

# API is now reachable at http://localhost:8000
```
