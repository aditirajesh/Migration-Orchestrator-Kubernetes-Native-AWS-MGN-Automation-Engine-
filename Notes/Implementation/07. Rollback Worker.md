#### When an MGN worker or a poller worker "nacks" a job with reenque = false (indicating the job could not be processed) -> it goes to the DLX queue which in turn forwards it to the mgn_jobs.failed and poll_jobs.failed queues . These will contain the original failed mgn and polled jobs. So info about which job failed, for which server and the payload for the same is known. 

## Core Mechanism:
The rollback worker doesn't reverse the history of calls made. Instead it reads the current state, maps it to a set of AWS undo actions it has to perform and does the action accordingly 

- ONLY UNDO IMMEDIATE FAILING STEP

![](../Images/Pasted%20image%2020260312112119.png)

**Note: stopping replication of a server is a risky move, which could lead to loss of data. Therefore only done with the replication is directly impacted. otherwise, none of the other rollback calls deal with stopping the replication of a server**

Additionally: a pre-flight check was implemented before launching test and cutover instance to make sure the replication was in a healthy state. If it is in a stalled/disconnected state, launching instances should be strictly prohibited. 

```
_get_lifecycle_state() helper — a single describe_source_servers call that returns the lifecycle state string. Both pre-flight checks use it so there's no duplicated API call logic.                   

_handle_launch_test_instance pre-flight — checks for READY_FOR_TEST before calling start_test(). Any other state — DISCONNECTED, NOT_READY, STOPPED, TESTING — returns a MgnResult.failed() with a message that clearly explains why the launch was aborted.                                                                                                                                             
_handle_start_cutover pre-flight — checks for READY_FOR_CUTOVER before calling start_cutover(). This is the more critical of the two — a production cutover against a lagging replication stream is the 
worst case scenario in any migration.    
```


## Components:

### 1. rollback_result.py:
- Rollback sends servers into two states:
```
  
ROLLED_BACK — every undo AWS call succeeded. The server will be marked FAILED — a known clean state an engineer can recover from.

FROZEN — an undo call failed, or no safe undo exists (cutover committed).  The server will be marked FROZEN — human intervention required before the automated system will touch this server again.
```

### 2. rollback_worker.py
This is the module that actually executes the rollback functions. It has a few functions defined in it:
- It initialises the list of terminal and always frozen states. For these states, the rollback worker simply has to acknowledge without performing any undo action 
- It connects to the rollback queue (via amql_url and channel "declare queue") and starts consuming the jobs within that queue 
- #get_rollback_plan is a plan used to get the full plan for rollback as well as the intended state it needs to go to. The intended state is either "failed" or "frozen", assuming the plan succeeds. 
- #execute_plan is the function that actually executes the rollback plan step by step
- #finalise is used to advance the state to failed or frozen depending on if the rollback was successful or not. 
- #on_message executes the full rollback workflow from start to finish. First it checks to see if the server is in a terminal/frozen states, gets the rollback plan, executes it and finalises accordingly.

### 3. run_rollback.py
- Entrypoint into the rollback worker
- This initialises the connection to the rabbitmq (provides the amql url), the database url for connection to postgres, creates the state manager client to update the transition history table as well as the servers table.
