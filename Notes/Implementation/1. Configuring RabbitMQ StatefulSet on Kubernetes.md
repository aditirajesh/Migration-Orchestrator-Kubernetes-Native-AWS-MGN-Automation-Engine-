
### 0. Install all dependencies:
- Docker
- Kubectl
- Kind (/minikube, but kind is better for multi-node cluster). This simulates a kubernetes cluster by running nodes as docker containers. 
- Helm 

### 1. Create cluster with kind config
```
  kind create cluster --config kind-cluster.yaml
  kubectl cluster-info --context kind-migration-orchestrator
  kubectl get nodes 

```
- CoreDNS and kubernetes control plane endpoints should be up

###  2. Create Secrets, Configs, Services
1. Secrets: username, password and erlang cookie. The username and password creates a default admin in the web admin page of RabbitMQ, and is used to help external clients connect with it (workers and APIs). The erlang cookie is an authentication mechanism that helps nodes talk to one another 
2. Config: 
   - 5672 port where the workers will listen to the queues for jobs.
   - 15672 separate port for the rabbitmq web admin.
   - #loopback_users is set to false to have connections inside the cluster
   - #rabbitmq_peer_discovery_k8s: used to help nodes auto-discover one another
1. Services:
   - Headless service: defined for the statefulset. This is used to give the pods an identifiable DNS name. Done with the RabbitMQ nodes to help them talk to one another
   - A regular service is defined as an endpoint for all other clients to talk to rabbitmq, and this appropriately load balances across all the rabbitmq nodes without exposing to which pod it lands on. 

### 3. Create StatefulSet 
- Here, the **volumeclaimtemplate** needs to be defined to create a PVC. This ensures that in the case where the pod shuts down, the data is retained and reattached to the new node with the same DNS name. This is what helps each node gets its own dedicated PVC.
- In the container, the rabbitmq username and pass creates a web admin. Volumes are mounted to store the config data, and the enabled plugins. 

### 4. Verify 
- Once all the templates have been executed, verify if a statefulset has been properly created with the proper configuration 
```
- k apply -f [filename]
- k get pods -n [namespace] -w (see if pod is up )
- k get pvc  -n [namespace] (see if PVC has been created for the statefulset pod)
-   kubectl exec -it rabbitmq-0 -n [namespace] -- rabbitmq-diagnostics status (check if rabbitmq is successfully executing inside of the pod.)
```

### 5. Access RabbitMQ Management UI
- To access the rabbitMQ management UI, the k port-forward command has to be executed for port 15672 or whatever port you have configured to host the management UI 
- Open http://localhost:15672 and login to the UI with the username and password credentials set in the secrets.yaml file. 

### 6. Create the queues
- Create three queues: #mgn_jobs, #poll_jobs and #rollback_jobs. All of them should be durable and should have x-dead-letter-exchange set = #dlxmigration. The dead letter exchange is used to re-route messages that could not be processed normally. Without it, these unprocessed messages will be slightly dropped with no way to trace or to rollback. With the dx-migration exchange, these messages can be rerouted to a dead letter queue and later tackled by the rollback worker. 
- Once the queues are created they are in an idle state with 'D' and 'DLX' feature tags indicating durable and dead-letter-exchange is set.

### 7. Dead letter exchange mechanism 
- In the UI, create a durable exchange that handles all failed messages -> call this dlx.migration. 
- ![[img9.png]]
- Create three more queues, mgn_jobs.failed, poll_jobs.failed and rollback_jobs.failed. These queues will hold the failed messages and need to be bound to the dlx.migration exchange. No arguments needed, make the queues durable. 
- Bind the .failed queues with the dlx.migration exchange, with the routing key = the type of job it is (mgn_jobs/poll_jobs/rollback_jobs)
